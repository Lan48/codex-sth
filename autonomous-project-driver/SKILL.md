# Autonomous Project Driver

## 概览

在单次用户请求内持续推进当前项目。先做一次轻量路由判断：如果仓库已经接入 GSD 且存在 `.planning/ROADMAP.md`，优先交给 `$gsd-autonomous` 处理；否则按通用自治模式自行判断下一步、直接实现并验证。把状态同步与请求许可分离；只在真正需要人工判断、外部信息或风险授权时暂停。

## 执行模式选择

1. 优先检测 `.planning/ROADMAP.md` 是否存在。
2. 如果存在：
   - 将当前请求视为 GSD 项目自治推进请求。
   - 优先使用 `$gsd-autonomous`，不要在本 skill 内重复实现 phase 级 discuss→plan→execute 流程。
   - 将用户当前意图、范围限制和显式 flags 原样传递给 `$gsd-autonomous`。
3. 如果不存在：
   - 退回本 skill 的通用自治推进流程。
4. 如果存在 `.planning/` 痕迹但没有 `ROADMAP.md`：
   - 不要假设 GSD 状态完整。
   - 继续按非 GSD 项目处理，除非仓库中还有更强的事实信号足以表明应走 GSD 流程。

## 自治合同

- 将“继续推进”“自动往下做”“不用每一步等我”“直到需要我介入再停”视为显式授权。
- 把一次回复理解为一个工作循环的阶段汇报，不是默认收工信号。
- 每完成一个可验证子任务后，立即重新判断下一步；除非命中暂停条件，否则继续执行。
- 进度更新可以短而频繁，但不要把更新写成“是否继续”的请求。
- 若任务本质只是计划、评审、文档建议或头脑风暴，不要擅自写代码。

## 建立上下文

1. 先确认最高优先级信号：
   - 用户当前消息中的明确目标和限制
   - 失败测试、报错、运行日志、CI 线索或 review comment
   - 路线图、todo、issue、`.planning/STATE.md`、`.planning/ROADMAP.md`
   - 相关文件中的 `TODO`、`FIXME` 或占位实现
   - 刚完成改动周边尚未闭环的验证、测试、文档、集成工作
2. 只读取最小必要范围：
   - 优先 `rg`、`rg --files`、`sed -n` 和定点读取，不盲读整文件
   - 先定位，再展开局部上下文
3. 若存在多个信号，按“离用户目标最近、风险最高、最能解锁后续工作”的顺序排序。
4. 若仓库存在 `.planning/ROADMAP.md`，优先路由到 `$gsd-autonomous`。
5. 若未命中 GSD 路由，但仓库存在 `.planning/STATE.md`、todo、issue 或同类执行痕迹，再把它们当作推进顺序的事实来源，而不是临时发明新顺序。

## 选择下一步

- 选择能实质推进目标、且当前信息足够支持实施的最小闭环任务。
- 优先补全当前工作线程，而不是在多个方向做表面小修。
- 默认顺序是：修复阻塞、让最近的改动真正可用、补测试或验证、清理明显技术债、再进入下一条 roadmap 或 todo。
- 如果可以可靠推断下一步，就不要停下来等待下一条指示。

## 执行与验证循环

1. 直接实施下一步。
2. 先跑最小相关验证，再视情况扩大到更完整的检查。
3. 验证失败时先自行排查并继续，除非失败暴露了暂停条件。
4. 每轮结束都重新回答两个问题：
   - 现在是否已经产生新的、明确的下一步？
   - 如果继续做，会不会跨入需要人工拍板的区域？
5. 只要两个问题的答案分别是“是”和“不会”，就继续下一轮，而不是结束回合。

## 必须暂停并询问用户的条件

- 需要产品、交互、架构或优先级决策，且不同选项代价或结果明显不同，仓库里又没有既定偏好。
- 需要密钥、凭证、账号、付费操作、生产权限、外部系统授权或法律合规确认。
- 需要执行破坏性或不可逆操作，例如删除大量数据、强推、发布、部署、迁移、覆盖用户现有内容，而用户未明确授权。
- 发现用户已有改动或既有指令与当前推断冲突，且无法安全合并。
- 已耗尽可靠信号源，存在多个合理方向，但没有足够依据判断哪条更优。
- 同一阻塞在多次认真尝试后仍未突破，需要用户提供外部信息或重新定方向。

## 沟通方式

- 用简短 commentary 同步：正在查什么、准备改什么、验证结果如何、为什么继续。
- 不要在中间节点用 final 提前收工；只有在完成、自然里程碑、或命中暂停条件时再给最终总结。
- 当需要提问时，先说明已推进到哪里、为什么这里必须人工介入，再提出一个尽量短的问题。
- 若必须停下，同时给出你推荐的默认方案和原因。

## 完成定义

满足以下任一情况时，才把当前回合作为完成：

- 用户目标已经实现并完成了合理验证。
- 已推进到一个自然里程碑，且下一步需要更大的上下文切换或新的用户目标。
- 命中暂停条件，已清楚说明卡点、已做工作和建议的下一步。

## 示例触发语

- `使用 $autonomous-project-driver 持续推进当前项目，直到真的需要我决定。`
- `用 $autonomous-project-driver 别每做完一步就停，能判断下一步就继续。`
- `使用 $autonomous-project-driver 把当前 backlog 往前推，遇到外部依赖或方案分歧再问我。`
- `使用 $autonomous-project-driver；如果检测到 .planning/ROADMAP.md 就优先走 $gsd-autonomous，否则按通用流程继续。`
